# prettyout Redesign — Design Spec

**Date:** 2026-05-14

## Goal

Pretty, grouped output for CLI tools that support JSON output. Transparent to the user: install once, tools work as usual, output is beautiful. Extensible: anyone can add custom formatters or override existing ones.

---

## Core Concepts

- **Registry** — describes every supported tool: which subcommands to intercept, JSON flags, passthrough flags, which launchers to wrap.
- **Config** — user preferences: enabled tools, plugin overrides, formatter settings, custom tools.
- **Hook** — shell functions generated from registry + config at shell startup. Subcommand-aware and launcher-aware.
- **Plugin** — any executable that reads JSON from stdin and writes pretty output to stdout.

---

## Registry (embedded YAML)

Built into the binary via `go:embed`. Users cannot edit this file directly; they extend via `custom_tools` in their config.

```yaml
tools:
  ruff:
    plugin: prettyout-ruff
    intercept_subcommands: [check]
    json_flags: [--output-format=json]
    passthrough_flags: [--watch, -W]
    launchers: [uvx, pipx]

  basedpyright:
    plugin: prettyout-basedpyright
    json_flags: [--outputjson]
    launchers: [uvx]
```

**Fields:**
- `intercept_subcommands` — only these subcommands are intercepted. Empty/absent = intercept all.
- `passthrough_flags` — if any of these flags appear in args, skip interception entirely (e.g. `--watch` streams output, can't buffer).
- `launchers` — list of launchers that should also wrap this tool (see Launcher section).

## User Config

`~/.config/prettyout/config.yaml`

```yaml
enabled:
  ruff: true
  basedpyright: true

# Replace a formatter entirely (any executable: binary, Python script, shell script)
plugins:
  ruff: ~/scripts/my-ruff-formatter.py

# Configure an existing formatter
settings:
  ruff:
    group_by: rule        # "rule" | "file"
    colors: true
    max_message_length: 100

# Register a custom tool not in the built-in registry
custom_tools:
  mycooltool:
    plugin: ~/scripts/mycooltool-fmt
    intercept_subcommands: [lint]
    json_flags: [--json]

# CI / non-TTY behavior
ci_mode: auto             # "auto" | "always" | "never"
                          # auto = if stdout is not a TTY, skip interception entirely
                          #        and run the original tool unchanged
                          # always = pretty output even in CI/pipes
                          # never = always pass through (disable prettyout globally)
```

**Per-project config:** `.prettyout.yaml` in the project root overrides the global config for that project. Same schema, all fields optional.

## Shell Hook

Generated by `prettyout hook <shell>`, eval'd at shell startup. All logic is baked into the generated functions — no subprocess calls at runtime except a single `prettyout _enabled <tool>` check.

### Direct tool wrapper (subcommand-aware)

```bash
ruff() {
  if prettyout _enabled ruff 2>/dev/null; then
    case "${1:-}" in
      check)
        for arg in "$@"; do
          case "$arg" in --watch|-W) command ruff "$@"; return $? ;; esac
        done
        local _errfile; _errfile=$(mktemp)
        command ruff "$@" --output-format=json 2>"$_errfile" | prettyout-ruff
        local _ret=${pipestatus[1]}
        cat "$_errfile" >&2; rm -f "$_errfile"
        return $_ret ;;
    esac
  fi
  command ruff "$@"
}
```

Key properties:
- **Stderr is separated**: stdout goes through the formatter, stderr is passed directly to the terminal. `2>&1` is not used.
- **Passthrough flags are checked**: if `--watch` is present, the original command runs unchanged.
- **Exit code is preserved**: from the tool, not the formatter.

### Launcher wrapper

Launchers (uvx, npx, pipx) are wrapped with argument-parsing logic that skips launcher-own flags to find the tool name.

```bash
uvx() {
  # skip uvx's own flags to find the tool argument
  local _tool_arg=""
  for _a in "$@"; do
    case "$_a" in
      --no-cache|--no-project|--isolated|-q|--quiet) continue ;;
      --python|-p|--with|--from) continue ;;  # these consume next arg too, handled below
      -*) continue ;;
      *) _tool_arg="$_a"; break ;;
    esac
  done
  local _toolname="${_tool_arg%%@*}"   # "ruff@0.5.0" → "ruff"

  case "$_toolname" in
    ruff)
      if prettyout _enabled ruff 2>/dev/null; then
        # find subcommand: first non-flag arg after tool
        local _past_tool=0 _subcmd=""
        for _a in "$@"; do
          if [[ "$_past_tool" == "1" ]] && [[ "$_a" != -* ]]; then
            _subcmd="$_a"; break
          fi
          [[ "${_a%%@*}" == "ruff" ]] && _past_tool=1
        done
        case "$_subcmd" in
          check)
            for _a in "$@"; do
              case "$_a" in --watch|-W) command uvx "$@"; return $? ;; esac
            done
            local _errfile; _errfile=$(mktemp)
            command uvx "$@" --output-format=json 2>"$_errfile" | prettyout-ruff
            local _ret=${pipestatus[1]}; cat "$_errfile" >&2; rm -f "$_errfile"; return $_ret ;;
        esac
      fi ;;
  esac
  command uvx "$@"
}
```

Launcher models in registry (drives hook generation):
```yaml
launchers:
  uvx:
    skip_flags: [--no-cache, --no-project, --isolated, --python, -p, --with, --from]
    tool_position: first_non_flag
  pipx:
    skip_flags: [--python, --spec, --index-url]
    prefix_args: [run]          # "pipx run <tool>" — skip "run" before tool name
    tool_position: first_non_flag
  npx:
    skip_flags: [--yes, -y, --no, --legacy-peer-deps, -p, --package]
    tool_position: first_non_flag
```

## Plugin Contract

A prettyout plugin is **any executable** (Go binary, Python script, shell script) that:
- Reads JSON from stdin
- Writes formatted text to stdout
- Exits with code 0 on success, non-zero on failure

Plugin resolution order:
1. User override in config (`plugins.ruff`)
2. `prettyout-<toolname>` found in `$PATH`
3. Error: no formatter found

## pkg/formatter API

```go
// Basic: read all stdin, call transform
formatter.Run(func(data []byte) error { ... })

// With config: reads ~/.config/prettyout/config.yaml + .prettyout.yaml
formatter.RunWithConfig(func(data []byte, cfg formatter.Config) error { ... })

type Config struct {
    GroupBy          string         // "rule" | "file", default "rule"
    Colors           bool           // default true
    MaxMessageLength int            // 0 = unlimited
    Extra            map[string]any // tool-specific settings
}
```

Config is loaded from global + per-project files, merged with per-project taking precedence.

## What Changes vs Current Code

| Current | New |
|---|---|
| Registry hardcoded in Go | Embedded YAML, user-extensible |
| Hook appends JSON flags to all subcommands | Subcommand-aware: only intercept listed subcommands |
| `2>&1` — stderr mixed into formatter | Stderr separated, passed directly to terminal |
| No launcher support | uvx, pipx, npx wrappers generated from registry |
| `backend/` hardcoded in plugins | Removed entirely |
| `registry.yaml` unused by Go code | `registry.yaml` stays as community index; built-in tool defs move to `internal/registry/builtin.yaml` |
| No formatter config | `pkg/formatter.Config` + per-tool settings in user config |

---

## Out of Scope (Not in This Design)

- Streaming formatters (tools with `--watch`)
- IDE integration
- Tool version compatibility / JSON schema versioning
- TUI (`prettyout tui`)
- `prettyout search` / community plugin discovery
